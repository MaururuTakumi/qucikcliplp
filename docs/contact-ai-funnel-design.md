# honkoma 問い合わせ動線 + AIチャット「AIに聞いてみる」設計書

> 設計: Fable 5（頭脳／IP担当）。実装: Opus以下。
> スコープ: 設計のみ。既存資産（tokens / ArrowCTA / inverseテーマ / GAS Contact）と整合。
> 中核: **「冷たいフォーム」を「AIに自社を診断してもらう体験」に置換する。** honkomaはAI伴走会社＝ボット体験そのものが最大の営業資料。訪問者がURLとメアドを"差し出したくなる"のは、その場で「御社ならこう使える」という一次価値が返るから。
> 関連: [component-animation-blueprint.md](./component-animation-blueprint.md) / [lp-redesign-brief.md](./lp-redesign-brief.md)

## 0. 設計の背骨（1文）
**「URLを入れる → その場でAIが御社サイトを読んで3つの活用案を返す（無料・即時）→ 精度の高い"AI活用診断レポート"はメールで送る」** という **価値先出し→マイクロコミット→価値交換** の3段で、リードを"お願い"せず"獲得"する。

## 1. 導線マップ（配置とファネル）

### 1.1 接触点（主導線1／副導線4）
| # | 配置 | 役割 | 起動UI | 優先 |
|---|---|---|---|---|
| **A** | Hero直下の専用インライン起動ブロック（"AI活用診断"帯） | **主導線**。URL入力欄が最初から見える（ゼロクリック着手） | URL入力＋ArrowCTA(fill/lg)「AIに聞いてみる」 | ★主 |
| B | フローティング起動ピル（右下・全ページ常駐） | 副。スクロール後もいつでも呼べる。FloatingNavと競合しない右下 | 円形ピル＋鼓動マイクロアニメ | ★副 |
| C | 事業(What We Do)末尾 | 文脈起動「自社だとどれが効く？」 | ArrowCTA(outline)「自社での使い方をAIに聞く」 | 副 |
| D | 実績/事例末尾 | 社会的証明の直後に着手（"うちもできる？"の熱を受ける） | インラインURL入力（A縮小版） | 副 |

- **全起動点が同じ ChatDrawer を開く**（入口5・体験1＝保守性/計測一貫）。各起動点は `source`(hero/float/whatwedo/cases/exit) をチャットに引き継ぎリード時に記録。

### 1.2 ファネル
```
[認知] 起動点A–E → [着手] URL入力/クリック(マイクロコミット①) → [一次価値] AIがURL解析→3案を即表示(無料・メアド不要)
 → [深掘り] 1–2問の対話(②) → [価値交換] "AI活用診断レポート(PDF)をメールで送る"→メアド取得(③)
 → [リード確定] メール検証→格納→honkoma通知＋自動レポート送付 → [商談化] "担当と話す"→Contact/日程調整
```
各離脱点に受け皿: 着手前離脱→exit-intent(E)/フローティング(B)常駐、メアド渋り→「3案の共有リンク」ソフトリード。

### 1.3 既存Contact(GAS)との関係
- 棲み分け: ContactPage(GAS)=「もう相談を決めた人」の明示的窓口(下部/フッター)。AIチャット=「まだ迷う人」を診断体験で温めるリード獲得エンジン。上流=チャット、下流=Contact。
- 統合: チャットのリード送信先は**GAS再利用(新パラメータ)でよい**（初期コスト低）。将来Notion/CRM化を見据え、格納は§5の抽象レイヤ経由。ContactからもチャットへクロスCTAを置き相互送客。

## 2. 会話フロー（発話スケルトン）

### 2.1 原則
価値を先・要求は後（メアドは3案表示の後）／1画面1アクション（プログレッシブ開示）／honkomaの"AIらしさ"体感（解析結果に訪問者サイトの固有名詞を織り込む＝"本当に読んだ"証明＝差別化の核）／常に前進が逃げより楽。

### 2.2 ステップ
- **Step0 着手(A/B/D)**: 「御社サイトのURLを入れてください。AIが見て honkomaならどこを自動化・売上化できるかを30秒で3つ挙げます（登録不要・無料）」→ 入力`会社URL`＋ArrowCTA(fill)「AIに聞いてみる」。※最初の要求が非個人情報のURL＝障壁最小(①)。
- **Step1 解析中(3–6秒)**: 「御社サイトを読み込んでいます…」→取得シグナルを逐次表示（「事業:○○を確認」「取扱商材:△△を検出」）。待ち時間を"本物のAI"証明に転化。不達は即フォールバック(§4.4)。
- **Step2 一次価値(3案・メアド不要)**: 「御社は{解析事業}とお見受けしました。honkomaなら— 1.{TOP LINE案} 2.{BOTTOM LINE案} 3.{FDE/内製化案}」。※提供価値3軸に必ずマッピング＝"会社の型"を体現。根拠に紐づけ。
- **Step3 深掘り(②・1–2問)**: 「今いちばん困っているのは？」→クイック選択チップ(人手不足/問い合わせ対応/データ活用/新規事業/不明)＋自由入力。タイピング最小で質情報取得。
- **Step4 価値交換(メアド・§4担保)**: 「{課題}に合わせて御社専用のAI活用診断レポート(PDF)を作成しました。受け取り先メールを入れてください。すぐ送ります」→`メール`＋ArrowCTA(fill)「レポートを受け取る」。※"作成しました"=sunk-value効果(③)。
- **Step5 確定＆商談化**: 「送信しました。内容について30分、担当と壁打ちもできます」→ArrowCTA「担当と話す」→日程調整/Contact。＋「3案の共有リンク」(社内稟議で拡散)。
- **分岐(Step4離脱受け皿)**: メアド渋り→「この3案の共有リンクをお作りします」→URLだけのソフトリード(source/URL/課題は記録済)＝ゼロにしない。

### 2.3 状態モデル
`idle→urlEntered→analyzing→insightsShown→qualifying→emailRequested→leadCaptured→handoff`（＋`analysisFailed`,`emailDeclined`分岐）。各遷移で計測イベント送出。

## 3. 離脱防止メカニクス（13策）
1.価値先出し(メアドは3案後に固定) 2.マイクロコミット階段(URL情報→1タップ選択→メアド) 3.プログレッシブ開示(1画面1アクション) 4.タイピング最小(選択チップ中心) 5.解析ローディングを"証明"に転化 6.sunk-value表現("作成しました") 7.exit-intent(PC)/スクロール80%(モバイル)・1セッション1回 8.フローティング常駐＋会話状態sessionStorage保持で復帰 9.モバイル最適化(ボトムシート・inputmode/type・キーボードで隠れない) 10.摩擦の言語化排除(無料/登録不要/30秒を明示) 11.中断安全(いつでも閉じ×戻れる・強制モーダルにしない) 12.フォールバック必須(誰もゼロで帰さない) 13.reduced-motion/a11y(減衰・focus trap・Esc・スクロールロック)。

## 4. リード取得の担保
### 4.1 なぜ正確な情報が出るか
- URL: 「AIに読ませる対象」＝間違えると自分が損する＝正入力インセンティブ内在＋到達性検証で即フィードバック。
- メアド: 「このメアドにレポートが届く」＝偽/捨てメールだと価値を受け取れない＝受領価値とメアド正確性を直結。B2Bは「社内共有しやすい会社アドレス推奨」を軽く添える。
### 4.2 価値交換
一次(無料・即時・メアド不要)=URL解析3案／二次(メアドと交換)=AI活用診断レポート(PDF・"作成済みを受け取る"フレーム)／三次(任意)=30分壁打ち/共有リンク。
### 4.3 検証
メール: 形式＋MX/ドメイン到達性(使い捨てドメイン照合)、不正は優しく差し戻し。URL: https正規化＋サーバ側HEAD/GET到達確認、不達は「事業を一言で」手入力フォールバック。二重送信/ボット: レート制限＋不可視honeypot/turnstile。
### 4.4 フォールバック(取りこぼしゼロ)
URL解析失敗→「事業を1行で」でAI診断続行／メアド拒否→共有リンク(ソフトリード記録)／AI失敗・timeout→「担当が直接見て後ほど送ります」＋メアドのみのミニフォールバックフォーム。
### 4.5 リードスキーマ
`source, companyUrl, analyzedSummary, proposedCases, painPoint, email, emailVerified, urlReachable, consent, timestamp, sessionId, utm/referrer`。※プライバシー同意チェックをメアド送信ステップに必須(PrivacyPageリンク)。

## 5. 技術アプローチ（設計レベル）

> **【2026-07-05 経営決定・§8参照】** LLM = **Grok API（xAI）**（Claudeではない）／基盤 = **Cloudflare Workers**（Vercelではない）／リード格納 = **Notion**（Leads DB）＋通知 = **Slack**。以下の比較・構成図中の「Claude API」は **Grok(xAI)** に、「Vercel/サーバレス関数」は **Cloudflare Workers** に、「GAS格納」は **Notion API** に読み替えること。

### 5.1 本物LLMボット（Grok/xAI） vs ガイド付き会話フォーム
| 観点 | 本物Claude API | ガイド付きフォーム |
|---|---|---|
| honkoma実力証明 | ◎最大の営業資料 | △見抜かれると逆効果 |
| URL解析パーソナライズ | ◎固有提案 | ✕事前定義のみ |
| 実装/運用コスト | 中(API・プロンプト・レイテンシ) | 低 |
| 暴走/幻覚 | 中(要ガードレール) | 低 |

**推奨: ハイブリッド。** 自由対話(Step1解析・Step2提案・Step3深掘りの言語生成)=**Claude API(本物)**＝最強の差別化かつ実力証明。リード取得の骨(Step4メアド・検証・格納・分岐)=**構造化フロー**でガチガチに固める(AIに任せない)。ガードレール: system promptで提供価値3軸に提案を制約・断定数値保証を禁止(景表法配慮)・社外秘/競合中傷/スコープ外遮断・出力長制限。

### 5.2 高レベル構成
```
[ChatDrawer(React,既存tokens/ArrowCTA)] → POST /api/ai-chat(session,source,url,message,step)
  → [サーバレス関数] (APIキーはサーバ側のみ・フロント露出禁止)
     ├(a) URL取得&サニタイズ→本文抽出
     ├(b) Claude API(system prompt=honkoma提案フレーム＋抽出本文) → 3案/深掘りをストリーミング返却
     ├(c) メール/URL検証(§4.3)
     └(d) リード格納レイヤ 抽象IF saveLead(payload)
           ├初期: 既存GAS(新パラメータ再利用) ├通知: honkomaへSlack/メール即時
           ├自動送付: レポートPDF生成→訪問者へメール └将来: Notion/CRMへ差し替え(IF不変)
```
- APIキー保護: サーバレス関数の環境変数のみ。Vite静的ホスティング想定なのでエンドポイントは別途サーバレス(Vercel/Cloudflare Functions等)を1枚噛ませる前提。
- レイテンシ: 解析→提案はストリーミング＋Step1ローディング演出でTTFT吸収。
- コスト/濫用制御: セッション単位トークン上限・レート制限・URL取得サイズ上限。
- モデル選定: 提案生成=上位モデル、検証/分類=軽量モデルで用途別(実装時 claude-api スキルでモデルID確認)。

### 5.3 レポート(PDF)
Claude生成の診断本文(3案＋課題別深掘り＋概算効果レンジ＋次アクション)をhonkomaブランドテンプレ(tokens配色・ロゴ)に流し込みPDF化・メール添付/リンク。**数値はレンジ・仮説表現に限定**(誇大回避)。

## 6. 既存デザイン資産との整合
ArrowCTA=全起動点の主ボタン(「AIに聞いてみる」fill/lg・副導線outline・レポート受取fill・外部/日程external+spin)。tokens/inverse=ChatDrawerを`data-theme="inverse"`(濃紺)で"特別なモード"として立て内部ArrowCTA/textが自動反転。モーション=ドロワー開閉ease.inOut+dur.base・メッセージ出現fadeUp(variants.ts)・ピル鼓動は控えめループ(reduced-motionで停止)＝新モーション資産の追加最小。フォント=見出しSpace Grotesk/本文Noto Sans JP。

## 8. 実装スタック確定（2026-07-05 経営決定）
以下で確定。§5の暫定比較・図はこれに読み替える。
1. **LLM = Grok API（xAI）**。本物のAIボットをMVPから入れる（ガイド付きフォームには落とさない）。honkomaはAI会社＝本物のAI体験が実力証明。
2. **基盤 = Cloudflare Workers**。Grok APIキー・各Secretは Cloudflare Secrets のみ（フロント露出禁止）。
   - ※静的LP配信は現状Vercel継続 or Cloudflare Pagesへ移行の別軸判断あり（backend=Cloudflareは確定）。CORS/構成簡素化の観点では Cloudflare Pages+Workers 一本化を推奨。
3. **リード格納 = Notion**（Leads用DB）へ `saveLead()` 抽象IF経由（Notion API + integration token）。将来他CRMへIF差替可。
4. **通知 = Slack Incoming Webhook**（新リードでチャンネル投稿）。
5. **レポート = MVPはHTMLメール**（3案＋診断）、PDFは後続強化。
6. 実装は Opus/Sonnet（fable不要・設計完了）。
