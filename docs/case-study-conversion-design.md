# 導入事例 → 問い合わせ動線 設計書（記事読了コンバージョン）

> 設計: Fable 5。実装: Opus。スコープ: **事例記事の読了者を自然に商談リード（AI診断 or /contact）へ乗せる導線のみ**。
> 前提整合: [site-ia-design.md](./site-ia-design.md) §3.5/§6（D導線 source="cases" は確定済み）、[contact-ai-funnel-design.md](./contact-ai-funnel-design.md)（チャット=上流/Contact=下流）、lp-redesign-brief（静かな確信・煽らない／監査#7/#14持ち越し禁止）。

## 0. 背骨（1文）

**事例は「証明」、CTAは「再現の誘い」。** 読了者の頭にある問いは「うちでもできるか？」— その問いにその場で答える装置（AI診断）を主、「もう決めた」人の出口（/contact）を副として、**記事の最終セクション直後に1ブロックだけ**置く。記事本文には売り込みを混ぜない。

## 1. 読了者心理 → 方針3行

1. B2Bで事例を最後まで読む人は **検討初期〜中期の情報収集者が大半**（稟議の材料集め・「自社に当てはまるか」の確認）。今すぐ商談したい人は少数派だが確実にいる（紹介経由・上申済み）。
2. 読中に営業要素を差し込むと**「お客様の声」の信憑性ごと割り引かれる**（記事が広告に見えた瞬間、証明力が死ぬ）。→ 読中CTAは置かない。
3. 読了直後（＝声を読み終えた直後）が熱のピーク。ここで「うちでもできるか？」に**その場で答える行動**（URL入力→30秒で3案）を最短で差し出す。押すのではなく、**問いの続きを引き受ける**。

## 2. 記事ページ `/case-studies/:slug` の導線設計

### 2.1 配置マップ

| # | 位置 | 要素 | 部品 | source | 主/副 |
|---|---|---|---|---|---|
| 1 | **記事末尾**（お客様の声＝最終セクションの直後、関連記事・戻るリンクより上） | 締めCTAブロック（§2.2） | `SectionShell` + `AIStarterBand` + `ArrowCTA` | `cases` | **主導線** |
| 2 | 全ページ共通 | フローティングピル（既存B・読中の受け皿はこれ**のみ**） | `FloatingChatLauncher` | `float` | 常駐 |
| 3 | exit-intent / スクロール80%（既存E） | `/case-studies/:slug` も発火対象に含める（IA§6-Eの「/case-studies」はslugページを含むと明記） | 既存E実装 | `exit` | 受け皿 |

**置かないもの（意思決定）**: 読中インラインCTA（§1-2）／記事専用のスティッキーバー（Bと競合・圧が出る）／「声」セクション内へのCTA埋め込み（証言と勧誘の同居は最悪の組み合わせ）。匿名事例など「声」が無い記事でも、締めブロックは最終セクション直後に同じ形で置く。

### 2.2 締めCTAブロック仕様（主成果物）

```
SectionShell (theme="default", wedge=false)  ← inverseにしない。記事の余韻のまま静かに置く
 ├ ENラベル: "AND YOUR CASE"（font-en・accent色、AIStarterBandのlabel流用）
 ├ 見出し(h2): §4のコピーA
 ├ 本文(1〜2行): §4のコピー（L1パーソナライズ差し込み・§3）
 ├ 主: AIStarterBand相当のインラインURL入力 + ArrowCTA(fill)「AIに聞いてみる」
 │    → useAiChat().startAnalysis("cases", url)（ChatDrawerが開き解析開始）
 └ 副: ArrowCTA(variant="outline", size="md", withText="導入の進め方を相談する", to="/contact")
      主の下に1行分離して静かに置く（横並びで対等に見せない）
```

- **主をインラインURL入力にする理由**: クリック→ドロワー→入力の2段を1段に潰す（#38 起動点Aと同じ「ゼロクリック着手」原理）。読了熱は数十秒で冷める。
- `AIStarterBand` は `compact` + `title`/`body` 上書きで再利用できる想定。h2階層が記事構造と衝突する場合のみ薄いラッパーを書く（見た目は同一に保つ）。
- inverse帯にしない理由: 濃紺の大CTA帯は「売り込みモード切替」のシグナルになり§1-2に反する。inverseはChatDrawer側（開いた後）の特別感に取ってある。

### 2.3 直後に置くもの（CTAの下）

締めブロックの下に「事例一覧へ戻る」`ArrowCTA(ghost, withText="導入事例一覧へ")` を1本。**逃げ道を残すことが「煽らない」の実装**であり、他事例の回遊＝信頼の積み増しは第二のコンバージョン経路。

## 3. パーソナライズ（"今読んだ事例"との接続）

### L1: コピー差し込み（必須・フロントのみで完結）

事例frontmatterに `industry`（業種名・表示用）と `serviceType`（AI導入/FDE/プロダクト開発）を必須フィールドとして持たせ、締めブロック本文に差し込む:

> 「**{industry}**のこの事例と同じ観点で、AIが御社サイトを読み、活用案を3つ返します。無料・登録不要です。」

`industry` 欠落時のフォールバック: 「この事例と同じ観点で、〜」（差し込みなしでも文が成立する構文にする）。

### L2: チャット冒頭への文脈引き継ぎ（任意・小改修1点）

`OpenOptions` を `{ source, companyUrl?, caseSlug? }` に拡張し、記事からは `caseSlug` を渡す。効果は2つ:
1. ChatDrawer冒頭挨拶を差し替え（「{事例タイトル}をご覧いただいた前提で、御社の場合を見てみます」）
2. `partial_lead` / lead payload に `caseSlug` を記録 → **どの事例が商談を生むかが計測できる**（コンテンツROIの一次データ）

拡張は型追加＋open時のstate保持＋payload転記のみで閉じる。Worker側スキーマ変更が重ければ `utm` 欄に `case:{slug}` を載せる代替でも可（Opus判断）。L1だけでも設計意図は成立する。

## 4. コピー（動詞終止・煽らない）

| 箇所 | A案（推奨） | B案 |
|---|---|---|
| 締め見出し | 同じ問いを、御社の現場に。 | この成果が御社で再現できるか、確かめる。 |
| 締め本文 | {industry}のこの事例と同じ観点で、AIが御社サイトを読み、活用案を3つ返します。無料・登録不要です。 | 会社サイトのURLを入れると、この事例と同じ診断を御社向けに30秒で行います。 |
| 主CTAボタン | AIに聞いてみる（サイト全体で統一・変えない） | 御社の場合を診断する |
| 副CTA（/contact） | 導入の進め方を相談する | 担当に相談する |
| 一覧へ戻る | 導入事例一覧へ | ほかの事例を見る |

- 主CTAボタンはA案固定を推奨: 「AIに聞いてみる」は全起動点共通の学習済みラベル（#38 §6）。記事だけ変えると認知コストが増える。文脈差は見出し・本文側で出す。
- 禁止: 「今すぐ」「限定」「無料診断実施中」等の煽り語（#14）、事例の数値を約束に転用する表現（「御社も◯◯%削減」— #7/景表法）。**事例の成果は過去の事実、CTAは未来の仮説**として言い切らない。

## 5. 2択の出し分け（AI診断 vs /contact）

- **主 = AI診断（source="cases"）**: 読了者の多数派（検討初期〜中期）に、メアド不要の一次価値が最も軽い次の一歩。視覚比重 8:2 目安（インライン入力フォーム vs 1行のoutlineリンク）。
- **副 = /contact**: 「もう決めた」少数派の直行出口。**必ず併置する**（決めた人にAI診断を強制すると一段遠回りさせて冷ます）。ただし横並びの2ボタンにはしない — 選択のコストを与えず、副は「下に静かにある」だけでよい。
- ChatDrawer内のStep5「担当と話す」→ /contact が既にあるため、AI診断に入った人の商談化経路は保証済み。二重にContactを推す必要はない。

## 6. 一覧ページ `/case-studies` 最下部

IA §3.5-#4 の確定どおり **置く**。ただし軽く:

- `AIStarterBand`（`compact`, `source="cases"`）1本のみ。title「御社の場合をAIに聞く」／body「事例と同じ観点で、AIが御社サイトから活用案を3つ返します。」
- /contact への副リンクは一覧では**置かない**（フッター直上でフッターのContactと重複するため）。記事側と役割分担: 一覧=軽い着手、記事末尾=二段構え。

### 受け入れ基準（実装チェック）

1. 記事本文（課題→取り組み→成果→声）の**内側にCTA要素がゼロ**であること
2. 記事末尾ブロック: 見出し＋本文（industry差し込み）＋インラインURL入力(主)＋/contactリンク(副)＋一覧へ戻る、の5要素・この順
3. URL送信で `source="cases"` の ChatDrawer が解析開始状態で開くこと
4. 煽り語・数値約束が含まれないこと（§4禁止リスト）
5. 一覧最下部は AIStarterBand 1本のみであること
