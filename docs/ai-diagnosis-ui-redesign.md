## 1. ビジュアル刷新指針 —「夜明けの盤面」

### 1.1 診断の見た目を1文で

**「日の出前の紺の空に、診断が進むほど夜明けの光が差してくる、静かな盤面」。** ヒーローの日の出の空と同じ物語をチャットが引き継ぐ。装飾ではなく“進行”そのものをブランド表現にする——これが「作り込まれた診断ツール」の署名になる。AIっぽい演出（スパークル・ロボット・グラデ乱用・タイプライター風カーソル）は全面禁止。

### 1.2 現状の「ダサさ」の正体（診断）

| 症状 | 原因 | 処方 |
|---|---|---|
| 箱だらけで散漫 | summary箱＋proposal箱×3＋事例(破線)箱＋バトン(青地)箱が同格に縦積み | **1画面1カード原則**（§1.4）。区切りはヘアラインと余白で作る |
| 素っ気ない・安い | 全要素が同じ文字サイズ帯(0.84–1.05rem)・padding一律1.25rem・階層なし | タイポ階層を4段に固定（§1.5）、余白を段階化（§1.3） |
| 破線ボーダー・半透明の濁り | `dashed`借用、surface-raisedを透明度で薄めた濁色 | 破線全廃。raised面は**不透明**で使う |
| 「診断ツール」に見えない | 進行が見えない・どこにいるか分からない | 常設の**夜明けプログレスレール**（§1.6） |

### 1.3 レイアウト定数（全phase共通）

- **外殻padding**: デスクトップ `1.75rem 2rem`、モバイル `1.25rem var(--space-gutter)`。ヘッダーとボディで同じ左右paddingに揃え、縦のガイドラインを1本にする。
- **縦リズム**: 画面内の要素間隔は 3段階のみ — 密 `0.5rem`（ラベル↔本文）/ 中 `1rem`（要素間）/ 大 `1.75rem`（ブロック間）。中間値を作らない。
- **ヘアライン**: `1px solid color-mix(in srgb, var(--text-primary) 10%, transparent)`。区切りは原則これ。ボーダー濃度は 10%（区切り）/16%（入力・チップ輪郭）の2値のみ。
- **角丸**: パネル外殻 `--radius-lg`、カード `--radius-md`、入力・ボタン `--radius-sm`、チップ `--radius-pill`。混在させない。
- **影**: 影はパネル外殻の1枚だけ（現行の `0 24px 70px -24px navy` を維持）。**パネル内部に影は置かない**。

### 1.4 面（surface）の規律 — 1画面1カード原則

- パネル地は inverse の `--surface-base`（navy-900）。ドロワー/インライン/モバイル全て共通で inverse 運用（診断面＝夜の空、で固定。LPの明るい面との対比が「別の道具に入った」感を作る）。
- **1画面につき「カード」（raised面の箱）は最大1つ**。それは常に“いまの主役”に与える（3仮説カード群は例外的に3枚1組を1カードと見なす）。他は余白とヘアラインで区切る。
- raised面: `--surface-raised` を**そのまま不透明で**。color-mixで薄めない（濁りの根絶）。
- アクセント地（`--color-accent-soft` 系の色付き面）は**バトンと完了系にのみ**許可。質問・プラン表示に色地を使わない（青地の乱用が“安さ”の一因）。

### 1.5 タイポグラフィ階層（4段固定）

| 段 | 用途 | 指定 |
|---|---|---|
| T1 見出し | 各phaseの問い・restatement | `1.25rem / 700 / line-height 1.5`（モバイル1.15rem）。1画面に必ず1つ、必ず最上部 |
| T2 本文 | 説明・プラン本文 | `0.95rem / 400 / 1.85`、色 `--text-secondary` |
| T3 ラベル | kicker・stepのphase名・軸名・"Q2/3" | `--font-en 0.72rem / 500 / letter-spacing 0.14em / uppercase`、色 `--color-accent` |
| T4 注記 | 透明性文言・免責・補足 | `0.8rem / 1.7`、色 `color-mix(text-primary 55%)` |

強調は**太字のみ**（色変え強調は禁止・リンクとCTAだけが青）。番号は必ず `01 02 03` の font-en 2桁（/aiのSTEPS表記と統一）。

### 1.6 署名要素 — 夜明けプログレスレール（新設・最重要）

ヘッダー直下に常設する高さ**2px**のレール。
- 地: ヘアライン色。充填: `linear-gradient(90deg, var(--brand-600), var(--brand-400))`（inverse下では十分光る。primitives直参照はこの1箇所のみ許可——“夜明けの光”はブランド原色で固定したいため。嫌なら accent→accent-bright で代替可）。
- 充填率のマッピング: idle 0% → analyzing 8→30%（進捗行と同期して段階増）→ insightsShown(Q1) 45% → deepening Q2 60% / Q3 72% → focusBuilding 80→95%（同期増）→ focusShown 100%。以降（booking/email/completed）は100%のまま。
- 遷移: `width` を `dur.base / ease.out`。reduced-motion時はトランジションなしで即時反映。
- レール右端上に T4 で残量表示: 「あと2問」「あと1問」「まもなく完成」（§3で詳述）。
- 意味: **進行＝夜明け**。診断完了時にレール全体が明るい青で満ちる＝「夜が明けた」。completed画面の見出し下にもこのグラデを1本引いて呼応させる。

### 1.7 部品の統一仕様

- **チップ（選択肢）**: min-height `2.75rem`（実タップ44px確保）、padding `0.65rem 1.1rem`、radius-pill、輪郭16%。hover: 輪郭とテキストを accent に（`dur.fast / ease.soft`）。**選択済み**: 地 `--color-accent-soft`・輪郭 accent・文字 text-primary。選択肢が長文の質問（Q3 AI活用）はチップを**縦積み全幅の選択行**にする（pill形状のまま幅100%・左揃え）——折返しpillの醜さを排除。
- **エコーバック行**: 回答直後にチップ群が**選択1つを残して消え**、その下に T2 で「〜50名の体制を前提に組み立てます。」が `opacity 0→1, y 8→0, dur.reveal, ease.out` で入る。選ばれなかったチップのfadeは `dur.fast`。
- **入力欄**: 現行 `.aichat-field` の骨格維持。ただし地は不透明 raised、高さ `3.25rem`、フォーカスリングは現行仕様。URL入力はフィールド＋送信を**1行結合**（デスクトップ: 右端に埋め込みsubmit。モバイル: 縦積み維持）。
- **CTA**: 主CTAは常に `ArrowCTA variant="fill" withText`。1画面に fill は**1つまで**。副導線は text-button（accent色・下線なし・hoverで下線）。`.aichat-submit`（青ベタ角丸ボタン）はフォーム送信（URL/メール）にのみ使い、意思決定CTAには使わない——「進む」はArrowCTAの世界観で統一。
- **アイコン方針**: 装飾アイコンは**ゼロ**。許可は close の X、ArrowCTA内蔵矢印、完了チェック（§3の進捗行 ✓ はSVG1種を自作、lucide `Check` 可）のみ。
- **代表の実写**: バトンには代表の実写を**置かない**（パネル内では顔写真が小さくなりすぎて安っぽい。/aiの信頼帯と予約先で顔は見えている）。人間味は文言で出す。

### 1.8 モーション規律

- 使ってよい動き: opacity/yの入場（`dur.reveal / ease.out`、stagger は `stagger.tight`）、レール幅（`dur.base`）、チップhover（`dur.fast / ease.soft`）のみ。phase切替は**前phase残像を残さない**——ボディ全体を `opacity 0→1, y 12→0` で入れ替える（AnimatePresence mode="wait" 相当、exitは `dur.fast`）。
- 禁止: スケール強調・バウンス・無限ループアニメ（待機中のドット明滅のみ例外・§3）・パララックス。
- reduced-motion: 全入場を即時表示に、レールはトランジション無し、待機ドットは静止＋テキスト更新のみ。

## 2. 配置の推奨

### 2.1 結論

| 環境 | 推奨（1つ） | 形状 |
|---|---|---|
| デスクトップ（LP起動点A–E） | **右フル高ドロワー**（維持・ただし作り直し） | 右端固定・高さ100dvh・幅 `min(480px, 92vw)`・左辺のみ `--radius-lg`・右からスライドイン（`x:100%→0, dur.base, ease.inOut`） |
| モバイル（LP起動点A–E） | **全画面テイクオーバー** | 下から `y:100%→0` で 100dvh 全面。ヘッダー固定＋ボディスクロール＋sticky CTA（§6） |
| /ai | **インライン維持**（現行） | ページ内カード。URL送信でカード上端にスクロール固定（v2 §1.4どおり） |

### 2.2 理由（両論を経た上での決定）

- **中央モーダルを退ける理由**: 診断は2〜4分の継続タスク。中央モーダルは「閉じるか進むか」の二択圧が強く、背後を完全に遮蔽するため“LPの文脈（実績・代表の顔）を横目に安心する”動線が切れる。またA–Eの起動点が全ページに散っている以上、どこから開いても位置が一定な右ドロワーの方が学習コストが低い。
- **現行実装の真の問題は「右ドロワーですらない」こと**: いまは下寄せセンター520px・上角丸＝出自不明の形。**フル高の右ドロワー**に作り直すことで、(1) 縦スペースが約2倍になり focusShown の縦長問題の物理的余裕が生まれ、(2) 「サイトに常設された診断ツールの定位置」という佇まいが出る。overlayの暗幕は現行48%を**32%**に下げ、LPが薄く見え続けるようにする（遮蔽ではなく併走）。
- **モバイルで全画面にする理由**: 92vhボトムシートは背後が1割見えるだけで文脈価値ゼロのままキーボード出現で崩れる。全画面にして自前ヘッダー（閉じる・タイトル・レール）を持つ方が「単体サービス級」に見え、sticky CTAも安定する。
- ChatStage共有の制約は不変: 変わるのは殻（ChatDrawer / AiDiagnosisPageのラッパー）だけ。

## 3. 離脱対策 —「考えている間に逃さない」

離脱が起きる3点（質問で考え込む・生成待ち・focusShownの情報過多）に対し、以下を正典とする。

### 3.1 質問を軽く見せる（deepening）

1. **1画面1問の徹底＋文脈の圧縮**: Q2/Q3では3仮説カードを**再表示しない**（現行は showThreeCards が deepening でも3枚出す→スクロールと情報量の主因）。代わりに画面最上部へ**回答レキャップ行**を1行だけ置く: 選択済みチップ（accent-soft地・小サイズ）で `問い合わせ・顧客対応` のように表示。「自分の入力が積み上がっている」ことが1行で見える。
2. **残量の明示**: T3ラベルで `Q2 / 3`、レール右上に「あと1問」。質問見出しの直下に置く helperテキストは1行まで。
3. **スキップの見せ方**: text-buttonのまま質問見出しと**同じ画面内・チップ群の直下**に常置（探させない）。文言は「スキップして次へ」（Q2/Q3）・「質問に答えず、このまま相談する」（Q1）。スキップを隠して答えさせる小細工はしない——答えない自由の可視化そのものが心理的軽さになる。
4. **回答の即時見返り**: チップタップ→エコーバック（§1.7）→ `600ms`（`dur.reveal`未満）置いて次の質問が入場。**無音の画面転換を作らない**。

### 3.2 待ち時間の演出（analyzing / focusBuilding）

1. **進捗行をチェックリスト化**: 現行の「点＋文」列挙をやめ、各行に3状態を持たせる — 完了行: 先頭 ✓（accent）・文字 text-secondary / 進行中行: 先頭ドット明滅（opacity 0.4↔1、`dur.reveal`往復、reduced-motionで静止）・文字 text-primary / 未来行: 40%透明で**先に薄く見せる**。「何が残っているか」が見えると待ちは短く感じる。
2. **実データの断片を流す**: analyzing中、行の完了に合わせて取得済みシグナル（`state.analysis.signals`相当が届き次第）を T4 で1〜2個先出しする。「本当に読んでいる」証拠が最強の引き止め。
3. **focusBuilding は“あなたの回答で作っている”を見せる**: 画面上部に回答レキャップ行（Q1×Q2×Q3の選択チップ並び）を置き、その下に進捗チェックリスト。見出しは「{課題}に絞って、組み立てています。」——待ち画面がパーソナルであるほど離脱しない。
4. **長引いた時の一言**: 8秒経過で T4 を1行追加「もう少しです。公開情報が多い会社ほど、少し時間がかかります。」（言い訳ではなく品質の演出に転化）。
5. **最低表示時間**: 進捗行1行あたり最低 `dur.reveal` 確保（一瞬で全部✓になるチープさの防止）。逆に全完了後の遅延ゼロで次phaseへ。

### 3.3 focusShown の情報圧縮（§6-6で構成を確定）

- ファーストビュー（スクロールなし）に入れるのは **restatement＋3ステップ＋主CTA** の3つだけ。役割分担・前提・リスクは折りたたみ、事例はコンパクト表示＋展開。
- **CTAはstickyフッター化**: focusShown以降、パネル下端に固定バー（上ヘアライン・地はsurface-base不透明）: 左に `ArrowCTA fill sm withText「弊社代表に相談してみる」`、右に text「メールで受け取る」。**スクロール位置に関係なく常にCVできる**。デスクトップ・モバイル共通（v2のモバイルsticky仕様を全環境に拡張）。

### 3.4 再開の保証

sessionStorage復帰（既存）を活かし、ドロワー再オープン時に保存phaseから再開＋ヘッダー下に T4 で「前回の続きから再開しています。<text-button>最初からやり直す</text-button>」を1行。

## 4. メール取得タイミングの推奨

### 4.1 結論: **(d) 段階的 —「予約クリック直後」を主取得点に追加する**

- **プラン提示前（選択肢b）は不採用**。深掘り3問は「診断を良くするための質問」という契約で成立しており、その直後・成果物の直前にメールを要求すると契約違反に見える（＝価値の出し惜しみ）。focusShown到達率を毀損し、v2の「連絡先は人間バトンまで要求しない」原則にも反する。
- **主取得点 = bookingStarted画面（予約クリックの直後）**。理由: (1) 予約タブを開いた瞬間が意思の最高点で、追加1フィールドのCVコストがほぼゼロ。(2) 「診断メモの送付先」という**利得つきの自然な名目**がある。(3) 診断コードとメールが揃うことで、予約突合（v2 §6.3の人力運用）の精度が上がる＝営業実務も救う。(4) 予約タブで離脱して予約未完了でも、メールが残れば翌日フォロー可能——社長の「予約に流れた人からメアドが取れていない」を正面から解く。
- **副取得点 = focusShownバトンの「メールで受け取る」**（現行維持・pull型・セッション1回）。
- 取らない点: プラン生成前・Q1〜Q3の途中・完了後の再要求。declined後の再要求なし（v2原則維持）。

### 4.2 動線図（取得ポイント=◉）

```
idle → analyzing → 3仮説+Q1 → deepening(Q2/Q3) → focusBuilding → focusShown
                                                                    │
                                     ┌──────────────────────────────┤
                                     ▼ 主CTA                        ▼ 副導線
                              bookingStarted                  emailRequested ◉②
                              （予約タブ＋診断コード）          「診断をメールで受け取る」
                                     │                               │
                              ◉① 送付先メール(任意)                  │
                              「診断メモの送付先」                    │
                              入力＋同意＋送信                        │
                                     │                               │
                                     └───────────→ leadCaptured ←────┘
                                                        │
                                                  enriching → completed
                                     （◉①をスキップしても bookingStarted
                                       →閉じる、は従来どおりpartial扱い）
```

### 4.3 ◉①（bookingStarted内・任意）の仕様

- 位置: 診断コード表示ブロックの**直下**（§6-7）。見出しなし、T2の1行＋1フィールド＋同意＋送信のみの最小フォーム。
- 文言: 「この診断メモを、メールにも残せます。予約と同じメールアドレスなら、当日の引き継ぎもスムーズです。（任意）」
- 同意: プライバシーポリシー同意チェックは**このフォーム内に必須**（emailRequestedと同一仕様・未同意はemail不搭載の構造保証を共有）。
- 送信成功 → leadCaptured系の確認は**同画面内に1行差し込み**（「送信しました。数分で届きます。」）で、画面遷移させない（予約が主タスクなので奪わない）。stage/contactMethodの記録は v2 §6.2 に従い `contactMethod: "booking"` を保持（メールも取れた場合は両方記録できるようフィールドは独立）。
- しつこくしない原則: ◉①を無視して閉じても、再オープン時に再提示しない（表示はセッション1回）。

### 4.4 トレードオフの明示（社長への説明用）

- 予約CVへの影響: ◉①は予約クリック**後**のため、主CV（予約）への摩擦追加は理論上ゼロ。増えるのは「予約意思を見せた人のメール捕捉率」のみ。
- 計測: `ai_booking_email_submitted`（新規・§7）で ◉① の取得率を独立に測る。低ければ文言調整、高ければ v2 の翌日フォロー運用（hot-booking Tier）が実データで回り出す。

## 5. コピー正典（温度統一版）

原則: v2 §2.1のトーン規定（敬体・静かな確信・感嘆符と希少性演出の禁止）を維持しつつ、**「代表を名指しで話す」重さを外す**。人名・役職の使い分けを次で固定する:

- チャット内の文言は「**弊社代表**」で統一（名指し・実写なし。軽さ優先）。
- /ai の信頼帯・FAQ・診断メールの署名では「代表・林拓海」を維持（静的な信頼情報としての実名はここに集約）。

| phase | 見出し(T1) / kicker(T3) | 本文・CTA |
|---|---|---|
| idle | 「御社ならAIをどう使えるか、診断します。」/ AI Diagnosis | 本文「会社サイトのURLを入れてください。登録不要・約60秒で、3つの活用仮説を返します。」／submit「AIに聞いてみる」／T4透明性文言（v2 §4のまま常設） |
| analyzing | 「読み込んでいます。」/ Analyzing | 進捗行3本（現行文言維持） |
| insightsShown | 「{companyName}への、3つの仮説。」/ 3 Hypotheses | Q1見出し「3つのうち、実情に近いものはありましたか。」helper「いちばん重い課題を1つ。ここから先の診断を、その課題に絞って組み直します。」／skip「質問に答えず、このまま相談する」 |
| analysisFailed | 同上＋注記(T4)「サイトの読み込みが不安定だったため、仮説ベースで続けます。壁打ちではその分、実情の聞き取りから始めます。」 | 誠実さをバトン理由に転化（v2 §2.4） |

## 6. 状態別レイアウト（fugu実装粒度）

共通骨格（ドロワー/全画面/インライン共通）: `[ヘッダー(kicker+T1+close)] → [夜明けレール] → [ボディ(スクロール)] → [stickyフッター(focusShown以降のみ)]`。ヘッダーとレールは固定、ボディのみスクロール。

**6-1 idle**: T1 → 本文1行 → URL入力（1行結合フォーム）→ T4透明性文言。要素4つだけ。中央寄せにせず左揃え・上から `1.75rem` 開始。
**6-2 analyzing**: レキャップなし。進捗チェックリスト（§3.2仕様）のみ。入力欄はフェードアウトして残さない。
**6-3 insightsShown / analysisFailed**: 解析サマリー（**箱を撤去**——T2本文＋signalsをT4の1行連結「読み取り: A ／ B ／ C」に圧縮・ヘアラインで上下区切り）→ 3仮説カード（唯一のカード群: raised不透明・radius-md・stagger.tight入場・軸ラベルはT3）→ ヘアライン → Q1ブロック（T1問い＋helper＋チップ6・「その他」タップで自由入力欄をチップ群の直下にインライン展開）→ skip text-button。
**6-4 deepening**: 3仮説カードを**出さない**。回答レキャップ行（§3.1）→ `Q2 / 3` ラベル → T1問い → helper1行 → チップ（Q2はpill横並び、Q3は全幅縦積み）→ エコーバック表示域 → skip。1画面が視界に収まること（スクロール発生させない）が受入条件。
**6-5 focusBuilding**: 回答レキャップ行 → T1「{課題}に絞って、組み立てています。」→ 進捗チェックリスト → （8秒超過文言）。
**6-6 focusShown**（縦長解消の本丸・上から）:
1. restatement（T1・1〜2行）＋ analysisFailed時のみ注記T4
2. 3ステップ: 縦3行のタイムライン。各行 `[T3 phaseラベル固定幅] [T2 action]`、行間 `1rem`、左に1pxの縦ヘアラインで結線。カード化しない。
3. `<details>`「体制と前提を見る」（T4トリガー・ヘアライン上下）: 中に roles（honkoma/御社）・prerequisites・riskNote。**デフォルト閉**。
4. 事例コンパクトカード（該当時のみ・本画面唯一のカード）: T3 kicker「近い状況では」→ タイトル(T2太字) → Outcome1行 → `<details>`「詳しく見る」内に Situation/Approach/期間 → 固定注記T4。次点事例は出さない（詳細内にも畳まない・情報量削減を優先）。該当なし時は1行「近い事例は、壁打ちで直接ご紹介します。」
5. バトンブロック（accent-soft地・radius-md・本画面唯一の色地）: 見出し＋本文（§5）＋agenda 2〜3点（T4箇条書き「壁打ちで確認すること」）。**CTAはここに置かず**stickyフッターへ。
6. stickyフッター（§3.3仕様）が同時に出現（`y 12→0, dur.base`）。
**6-7 bookingStarted**: T1 → **診断コードブロック**（raised不透明カード・コードは font-en `1.5rem / letter-spacing 0.14em` で大きく・右に「コピー」text-button）→ コード案内T2 → ヘアライン → ◉①メール最小フォーム（§4.3・送信後は同位置に確認1行）→ ArrowCTA outline「日程調整をもう一度開く」→ fallback text。stickyフッターは非表示に戻す。
**6-8 emailRequested**: T1 → 本文 → email入力 → 同意チェック → submit → decline。要素6つ・1画面完結。
**6-9 leadCaptured → enriching**: 確認見出し＋1行 → ArrowCTA outline「日程を選ぶ」→ ヘアライン → enrich 1問（チップ4＋skip）。色地は使わず通常面＋レール100%で静かに。
**6-10 completed**: T1「準備が整いました。」→ 見出し下に夜明けグラデ1本（§1.6呼応）→ 1行 → 「別のURLで診断する」。
**6-11 emailDeclined**: §5のとおり最小3要素。
