## 0. 判断の背骨（3行）

1. **ヒーローは「命令」から「観察」へ。** 現「時代を、自分事にしろ。」が不自然なのは、日本語の問題である以前に*態度*の問題 — 初対面の候補者に命令形で説教している。社長の言いたいこと（スキルで差がつかない時代、残るのは熱意）は本来*時代の観察*であり、観察として言い切るほうが強く、自然で、説教くさくならない。
2. **求人票は「無い」ことを弱点でなく設計として見せる。** 「募集職種: なし」を正面から書く。10名の会社が空欄の求人票を並べるより、「人物像4つ＋まず話す」に全振りするほうが誠実さが差別化になる（IA正典 §4 の思想を、社長の訂正＝インターン〜正社員幅広く・職種一覧なし、で更新）。
3. **動線は「基盤は統合・体験は分離」。** 採用エントリは問い合わせフォーム基盤（GAS→将来saveLead/Notion/Slack）に相乗りするが、**AI診断・営業ボイスは採用文脈に一切出さない**。最終的な主CVはGoogleカレンダー予約（新規タブ）に置き、フォームは受け皿に降格する。

## 1. ヒーロー（コピー案＋推奨）

### 1.1 構造（現行維持・中身だけ差し替え）

SectionShell(default) + SectionHeading(level=1) + リードp + CTA行。階層は **英字アクセント（enLabel）→ 主見出し → リード → CTA**。ホームヒーローと同じく英字は「訳文」ではなく「もう一つの声」として機能させる。

### 1.2 主見出し案（4案）

| 案 | 主見出し | 英字アクセント | 評価 |
|---|---|---|---|
| **A（推奨）** | **スキルの差は、AIが埋める。熱意の差は、埋まらない。** | *Skills converge. Passion doesn't.* | 対句で「言いたいこと」を丸ごと言い切る。命令形ゼロ＝説教にならない。「埋める/埋まらない」の対で記憶に残る。後半が熱意ある読み手への招待として反転する（＝あなたの熱意は無駄にならない、と読める） |
| B | AI時代、残るのは熱意だけ。 | *What remains is passion.* | 社長の温度感に最短・一行で軽い。ただし「なぜ残るのか」をリードに全部預けるため、一言の質量では A に劣る |
| C | スキルはAIが書ける。熱意は書けない。 | *AI can write your skills. Not your fire.* | Aの変奏。「書ける」がAI会社らしい語彙で鋭いが、「スキルを書く」の日本語がやや飛躍 |
| D | AI時代、あなたのスキルでは差がつかない。残るのは熱意だけ。 | *Your skills won't set you apart. Your passion will.* | 社長原文ほぼそのまま。二人称「あなたの」が刺さる人には刺さるが、初見の候補者への攻撃として読めるリスクがある |

**推奨: A。** 理由 — ①社長のメッセージ（スキルで差がつかない／残るのは熱意）を一言で*両方*言えている唯一の案。②サイトの声（「誰かが作った時代を、生きるな。」「会社を、AIネイティブへ。」）と同じ、読点で刻む対句リズム。③Dの二人称攻撃性とA旧コピーの命令形説教を両方回避し、「自分事」の精神は*観察の中身*（熱意＝自分事の度合い）として温存。

- **改行指定**: `title={["スキルの差は、AIが埋める。", "熱意の差は、埋まらない。"]}` の2行。モバイル(375px)で `--fs-hero` が折り返す場合は4分割 `["スキルの差は、", "AIが埋める。", "熱意の差は、", "埋まらない。"]` に落とす（実機確認で判断・受入基準参照）。
- **enLabel**: 現 `Careers` を上記英字アクセントに差し替え（ホームヒーローが英文をラベルに使う前例あり）。ページ判別は `document.title`（採用情報 | honkoma・変更なし）とナビが担う。

### 1.3 リード（推奨Aとセット・確定コピー）

> AIが誰の手にも届いた今、「何ができるか」の差は日ごとに縮んでいく。
> 最後に差になるのは、目の前の課題をどこまで自分事にできるか——つまり熱意だ。
> honkomaは、その熱を持つ仲間を探している10人のチーム。

- 3文構成: 観察（スキル差の消滅）→ 定義（熱意＝自分事の度合い。行動指針01への静かな接続）→ 自己紹介＋招待。「求人票の空欄を埋める採用はしない」の一文は §2 のセクションに移設（ヒーローを1メッセージに保つ）。
- 案B/Dを採る場合のリードも同骨格でよい（観察→定義→招待）。1文目だけ主見出しと重複しないよう調整。

### 1.4 CTA（現行踏襲・文言のみ確認）

- fill / lg: **「まず話す（カジュアル面談）」** → §3 の主動線（Phase 1: `/contact?type=recruit`、Phase 2: 予約URL）
- outline: **「メンバーを見る」** → `/team`

## 2. 募集方針・人物像セクション

### 2.1 セクション見出し（現行維持）

enLabel `Who We Look For` / 見出し **「職種ではなく、人で募る。」** — 変更なし。導入リードのみ差し替え:

> 求人票はありません。あるのは、この4行だけ。
> ひとつでも「自分のことだ」と思えたら、まず話しましょう。

### 2.2 人物像4項目（PERSONAS 差し替え・確定コピー）

社長の人物像4点（AIに興味／人と話すのが好き・少し明るい／コミュ重視・スキル必須でない／自分事）を、現行の一行リストから**「見出し＋一行本文」**に解像度を上げて置き換える。番号付きリストUI（01–04・borderTop行スタイル）は現行踏襲、本文用の小さめテキストを1行足す。

**01 AIに、心が動く人**
> 最新のAIを「仕事の話」としてではなく、「面白い話」として追いかけてしまう。その好奇心が出発点。

**02 人と話すのが、好きな人**
> honkomaの仕事は、現場の人との会話から始まる。少し明るい、はそれだけで立派な強み。

**03 スキルは、これからでいい人**
> PMも開発も、今できる必要はない。伝える力と学ぶ速さがあれば、道具はAIと私たちが揃える。

**04 仕事を、自分事にできる人**
> 頼まれた範囲で止まらず、「自分ならどうするか」で考える。行動指針の第1原則、そのまま。

- 04が行動指針01「自分事」への橋。直後に行動指針セクション（同文再掲・変更禁止）が既にあるため、ページ順は **人物像→行動指針** に入れ替えず現行順（ストーリー→行動指針→人物像）のままでよい。04の一文が後方参照として機能する。

### 2.3 分類3枠（TRACKS）の廃止 → 「かかわり方」ストリップに置換

現行の**エンジニア/ビジネス/オープンの3カードは廃止**（本書が IA §4-4 を上書き）。理由: 社長の訂正「形式的な職種一覧は作らない」に対し、3枠はミニ求人票として読まれる。代わりに「インターン〜正社員まで幅広く」を正面から言う一枚ストリップ（surface-raisedカード1枚 or 横並びテキスト）:

**見出し（h3相当）: かかわり方は、話しながら決める。**
> インターン、業務委託、正社員——入り口はどこからでも。
> 今のあなたに合うかかわり方を、面談で一緒に見つけるところから始めます。

- 3語（インターン/業務委託/正社員）は font-en 風の等間隔ラベル列などで軽く見せてよいが、**カード3枚に分けない**（枠＝応募区分と誤読させない）。
- 「正社員・インターン・業務委託が混ざり合い」は創業ストーリー02に既出であり、事実整合OK（正データ: 10名体制）。

### 2.4 セクション末尾に中間CTA（新設・軽量）

人物像の直後、離脱前の温度が最も高い位置に一行＋ArrowCTA(md/fill):

> ひとつでも心が動いたら、30分だけ話しましょう。
> CTA: 「カジュアル面談へ」→ §3 主動線

## 3. カジュアル面談の動線

### 3.1 動線の全体図

```
/recruit
 ├ Hero CTA「まず話す（カジュアル面談）」──┐
 ├ 中間CTA「カジュアル面談へ」────────────┼─→ [主動線]
 └ 末尾CTA（Let's talk セクション）───────┘
      Phase 1: /contact?type=recruit（種別プリセット済・実装済）
      Phase 2: Googleカレンダー予約ページ（新規タブ）★主CVに昇格
      副動線: /contact?type=recruit（フォームで連絡）
      第3動線: 代表X DM（@moriyorihayash1・現行維持）
```

### 3.2 末尾CTAセクション（inverse・確定コピー）

enLabel `Let's talk` / 見出し「まずは、話すことから。」（現行維持）。リード差し替え:

> 履歴書も、志望動機も、準備もいらない。
> 30分、オンラインで雑談するところから始めましょう。
> これは選考ではないので、身構えなくて大丈夫。

リード下に**事実の3点マイクロコピー**（font-en数字なし・小さめテキスト行）: `オンライン30分 ／ 選考ではありません ／ 準備するものなし` — 応募ハードルを下げる摩擦の言語化排除（contact-ai-funnel-design §3-10と同じ技法）。

CTA構成:
- **Phase 2（予約URL発行後）**: fill/lg **「面談日程をえらぶ」** = `href=RECRUIT_BOOKING_URL` + `direction="external"`（新規タブ・rel=noopener。ArrowCTAが target 未対応なら実装時に拡張）／ outline **「フォームで連絡する」** → `/contact?type=recruit` ／ ghost **「代表にXでDM」**（現行維持）
- **Phase 1（URL未発行の間）**: fill/lg **「カジュアル面談を申し込む」** → `/contact?type=recruit`（現行のまま）／ ghost X DM

実装は `HERO_HAS_VIDEO` と同じ**単一スワップポイント**方式: `const RECRUIT_BOOKING_URL: string | null = null;` — URLが入ったらCTA構成が自動でPhase 2に切り替わる分岐を1箇所に閉じる。

### 3.3 予約ページ（Googleカレンダー）の要件 — 社長タスク

- Googleカレンダーの**予約スケジュール（appointment schedule）**を1本発行: 枠30分・Google Meet自動発行・フォーム項目は名前＋メール（＋任意で「今なにをしている人か」1行）。
- 主担当は代表を推奨（「代表と30分」は10名企業にしかない吸引力。ページに「相手は代表です」と書けるかは要判断リスト④）。
- 予約発生はGoogleカレンダー通知で完結（Phase 3までSlack連携は不要・運用を増やさない）。

### 3.4 フォーム受け皿（現行 /contact の採用モード）

- 現行実装で `?type=recruit` → 種別「採用・カジュアル面談を希望する」プリセットが動作済み。**Phase 1はこのまま出荷できる（依存ゼロ）**。
- 送信後の期待値設定だけ改善余地: 採用種別のとき完了メッセージを「日程調整のご連絡をします（1〜2営業日目安）」系に差し替え——これは §4 の contact-funnel-v2 側要件に含める。

## 4. 問い合わせフォーム統合の判断と推奨

### 4.1 推奨: **統合（基盤共有）— ただし「体験は分離」**（完全別動線にはしない）

| 観点 | 統合（採用モード） | 完全別動線（/recruit専用フォーム） |
|---|---|---|
| UX | ◎ deep link 1本でプリセット・遷移1回 | ○ ページ内完結だが、体験の質はほぼ同等 |
| 運用（10名） | ◎ フォーム1本・GAS/Worker 1系統 | ✕ 検証・スパム対策・完了メール・保守が二重化 |
| リード管理 | ◎ saveLead 1本 → Notion 1 DB（type property で採用/商談を分岐）→ Slack 1系統 | △ 別DB・別通知の管理コスト |
| リスク | △ v2実装が営業ボイスを採用モードに漏らす懸念（→下の契約で封じる） | △ 更新が止まった専用フォームは死んだ入口になる |

**判断**: 完全別動線の利点は「営業色の遮断」だけであり、それは統合側でも**モード契約**で達成できる。10名の運用体制でフォーム2本は持たない（IA正典 §9「持てる器だけ持つ」と同判断）。さらに本設計では**最終的な主CVはカレンダー予約（フォーム外）**に移るため、フォームは受け皿——受け皿のために専用実装を立てる理由がない。

### 4.2 contact-funnel-v2 側への要件（採用モード契約 — v2設計・実装が守ること）

`docs/contact-funnel-v2-design.md` に以下を組み込むこと。**Opusは実装時に両設計を突き合わせ、矛盾があれば本節を優先**:

1. **URL契約の維持**: `/contact?type=recruit` で採用モード起動（現行挙動を後方互換で保持）。
2. **モードでの体験分離**: 採用モードでは (a) AI診断帯・AIチャットクロスCTA・診断誘導コピーを**非表示**（IA正典 §6「AIチャットは👤専用装置」と同一原則）、(b) 会社名・部署は非表示または任意、(c) 見出し・placeholder・完了メッセージを候補者向けボイスに差し替え（「御社」語彙禁止・完了時「日程調整のご連絡をします」）。
3. **任意フィールド1つだけ追加**: 「希望のかかわり方」select（インターン／業務委託／正社員／まだ決めていない・default=まだ決めていない）。必須にしない。
4. **リードスキーマ**: `saveLead` payload に `type: "recruit"` を追加（contact-ai-funnel-design §4.5 のスキーマ拡張）。格納は既存 Notion Leads DB に type property で同居、Slack通知は採用専用チャンネル or メンションで分岐（要判断リスト⑤）。
5. **非依存の明記**: v2が未着地でも現行 ContactPage の `?type=recruit` プリセットで /recruit は成立する。**/recruit 刷新は v2 をブロックしない・されない**。

## 社長への要判断リスト

1. **ヒーロー主見出し**: 推奨A「スキルの差は、AIが埋める。熱意の差は、埋まらない。」でよいか（対抗: B/C/D — §1.2）。
2. **Googleカレンダー予約ページの発行**: 30分枠・Meet自動発行で作成し、URLを共有（これがP2の唯一のブロッカー）。
3. **面談の相手**: 「相手は代表です」とページに明記してよいか（明記できると吸引力が一段上がる）。
4. **「選考ではありません」表記**: コピーどおり運用してよいか（面談を実質選考として扱うなら、この文言は下ろす必要がある——書いた以上は守る）。
5. **採用リードの通知先**: Slackの採用専用チャンネルを作るか、既存リードチャンネルにメンション分岐か（P3で必要）。
6. **X DM動線の存続**: 現行どおり `@moriyorihayash1` へのDM導線を残してよいか。
