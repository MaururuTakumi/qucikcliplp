## 0. なぜ今の出力は汎用なのか — 現行プロンプトの病巣（実装前に共有すべき診断）

社長指摘「読む気にならない」「AIが分かってる感がない」の原因は、モデルの能力不足ではなく**プロンプトが汎用出力を教えている**ことにある。`workers/ai-chat/src/index.ts` の該当箇所:

| # | 病巣 | 場所 | 何が起きているか |
|---|---|---|---|
| P1 | **analyzeのfew-shot例文自体が汎用文** | `deepSeekRequestBody` L566-591 | 例に「問い合わせ前の疑問に答える」「一次対応を自動化する」「現場運用に接続する」という**どの会社にも当てはまる文**を渡している。LLMは指示より例を真似る。汎用の見本を見せれば汎用が返る |
| P2 | **deepenの例に汎用フォールバックをそのまま渡している** | `deepSeekFocusPlanRequestBody` L1101 `example: fallbackFocusPlan(body, matchedCase)` | 「整理／試作／運用化」「Slack、Notion、フォームなど既存導線に合わせて小さな試作を作ります」への**最強のアンカー**。社長が見た「現状把握/最適化設計/運用定着」的出力の直接原因 |
| P3 | **「specificに」という抽象命令だけで、操作的指示がない** | 両system prompt | 「Keep Japanese concise and specific」の1行のみ。**何を**サイトから引用し、**どこに**織り込むかの手順がない。signalsの定義もない（→「問い合わせ導線を確認」のような作業報告になる） |
| P4 | **deepenにサイト由来の具体が渡っていない** | `deepSeekFocusPlanRequestBody` L1077-1102 | `DeepenBody.signals` は型に存在する（L269）のにDeepSeekへのuser contentに**含めていない**。deepenの燃料はanalyzedSummary1文だけ。具体の仕入れがなければ具体は出ない |
| P5 | **honkomaの提供価値（4本柱）がどちらのプロンプトにも存在しない** | 両prompt | データ基盤/共通AI基盤/BOAI/人は人、の語彙がゼロ。診断がhonkomaの提案にならないのは当然 |
| P6 | **フォールバックが完全汎用** | `mockAnalysis` L401-441 / `fallbackFocusPlan` L944-988 | ドメイン名の差し込みのみ。painやサイトテキストによる分岐は`firstAction`の1箇所だけ |

## 1. specificity / Barnum設計 — 占い原理の正しい使い方（捏造禁止との両立）

### 1.1 原理の転写を間違えない

占いが刺さる正体は「曖昧な一般論（Barnum文）」ではなく、**「見られている」という感覚**にある。曖昧な一般論をそのまま使えば、それこそが今の「汎用で読む気にならない」出力だ。転写すべきはBarnum文ではなく**コールドリーディングの構造**——①観測できた事実を自信をもって言い当てる（hit）→②信頼が立った状態で高確率の推測を当てにいく→③相手が自分で穴を埋める。

これを診断に落とすと、全ての「刺さる文」は次の**三段構造**で書く:

```
[観測]  サイトに実在する具体を言い切る          ← 言い切り。捏造リスクゼロ（見たものだけ）
[橋]    業界・事業モデルの一般論をつなぐ          ← 一般論として真であることだけ言う
[仮説]  御社の内側を当てにいく                   ← 必ず仮説形・質問形。「〜ではないでしょうか」
```

**例（架空の工務店・注文住宅とリフォーム、施工事例80件掲載、問い合わせは電話とフォームのみ、採用ページで施工管理を募集中）:**

> [観測] 施工事例を80件以上公開されていて、これは同規模の工務店ではかなり厚い資産です。
> [橋] 事例を読み込んでから問い合わせる層は、夜間や休日に温度が上がることが多い。
> [仮説] ですが窓口は電話とフォームのみ——**温まった見込み客を、営業時間の外で取りこぼしていないでしょうか。**

「なんで分かるの」の感覚は[仮説]で生まれるが、その信用は[観測]の的中が担保している。**言い切るのは観測、当てにいくのは質問。** この分業が「確信度の高い言い切り」と「捏造禁止」を両立させる唯一の解だ。

### 1.2 hit（観測）の採掘場 — サイトから何を拾うか

「サイトを読んだ」証明になるのは、**相手が書いたが、自分では診断材料と思っていない情報**。プロンプトに次の観測チェックリストを持たせる（analyze手順の第1歩・§2.2）:

1. **事業の構成**: 何を売っているか・事業が何本あるか・商材やサービスの固有名
2. **問い合わせ導線**: 電話のみ/フォーム/LINE/チャット——窓口の種類と数
3. **媒体・チャネルの散らばり**: EC+実店舗、LINE+Instagram+電話、複数ブランドサイト（→データ基盤柱の最重要トリガ）
4. **採用ページ**: 何の職種を募集しているか（＝どこの人手が足りないかの自白）
5. **コンテンツ資産**: 施工事例・症例・ブログ・FAQ・料金表の有無と厚み
6. **業界特有のオペレーション語**: 見積り、レセプト、車検、内見、予約枠…（業界の解像度を示す語）
7. **店舗・拠点数、対応エリア**
8. **更新の気配**: お知らせ・ブログの活性（※指摘の言い方は§1.4トーン規定に従う）

### 1.3 「当てにいく問い」の作り方

コールドリーディングでいう high-probability guess。**その業界・その事業モデルなら8割の会社が抱える痛み**を、観測に接続して質問形で出す。当たれば「うちを見てる」、外れても質問なので誤りにならず、「違うなら実情を聞かせてほしい」という壁打ちバトンの理由に転化する（v2 §2.4の誠実さ設計と同じ構造）。

| 観測されたら | 当てにいく問い（型） |
|---|---|
| 複数の問い合わせ窓口（LINE/電話/フォーム…） | 「どの媒体に何が来たか、今は担当者の記憶と個別アプリの中に分かれていませんか」 |
| 施工事例・症例・実績の厚い掲載 | 「この資産、問い合わせ対応や提案書には再利用できていますか」 |
| 採用ページで現場職を募集 | 「採用が埋まるまでの間、その業務は誰が巻き取っていますか」 |
| 料金表がない・「お見積もり」のみ | 「見積もり前の概算質問が、電話対応の時間を食っていませんか」 |
| EC+実店舗など複数チャネル | 「チャネル別の数字を、ひとつの場所で並べて見られていますか」 |
| ブログ/お知らせが活発 | 「書ける人が限られていて、その人の時間が上限になっていませんか」 |

### 1.4 捏造禁止との線引き（監査方針・景表法の死守ライン）

| 区分 | 扱い |
|---|---|
| **出してよい（言い切り可）** | サイトに実在する記載の引用・言及（社名、事業、商材名、掲載件数「施工事例80件」等の**相手自身の公開情報**）／業界の一般論（真である範囲）／honkomaの行為の宣言（「〜から始めます」） |
| **形を変えれば出せる（仮説形・質問形のみ）** | 相手の内部状態の推測（属人化してそう、データが散在してそう）／効果の見込み（「取りこぼしにくくなります」等、**数値なしの方向性**） |
| **絶対に出さない** | サイトにない事実の断定／効果の数値（「◯%削減」「月◯時間」——lint済みだがプロンプトでも禁止を明言）／裏取り無し定量／クライアント実名（lint辞書）／「必ず・確実に・保証」 |

**トーン規定（追加）**: 観測の指摘が粗探しに聞こえた瞬間に信頼は死ぬ。「古い」「不十分」「できていない」の断定は禁止。**欠点ではなく「余地」「もったいなさ」の語彙**で書く（「事例80件という資産が、問い合わせ対応に使われていないのはもったいない」）。感嘆符・煽りの禁止は現行lint規定のまま。

### 1.5 before → after（analyzedSummary）

**Before（現行の実出力レベル）**
> example.co.jp の公開情報をもとに、問い合わせ・集客・現場業務の改善余地を仮説として整理しました。

**After（三段構造）**
> ◯◯工務店様は、注文住宅とリフォームの2本柱で、施工事例を80件以上公開されているとお見受けしました。一方で窓口は電話とフォームのみ、採用ページでは施工管理を募集中——**事例で温まった見込み客の受け皿と、現場の人手の両方が細くなっていないでしょうか。**

## 2. 3案（analyze）プロンプト方針

### 2.1 生成手順を段階化する（プロンプト内で明示する思考手順）

「specificに書け」ではなく、**手順を踏ませる**。system promptに以下を日本語で明記（DeepSeekの日本語出力安定のため、systemも日本語主体に切り替える。JSONキーは英語のまま）:

1. **抽出**: websiteTextから§1.2チェックリストの観測を**最低3つ**拾い、`signals`に書く（後述の新定義）
2. **推定**: 業界と事業モデル（誰に・何を・どう売るか）を1行で特定する
3. **選定**: この業界・このモデルで「効く一点」を軸ごとに選ぶ（§2.3の軸再解釈表）
4. **生成**: 各proposalは三段構造（観測→橋→仮説）を圧縮して書く。**title・bodyのどちらかに、websiteTextに実在する固有語を必ず1つ以上含める**

### 2.2 signalsの再定義 — 「作業報告」から「hitの帯」へ

現行のsignals（「問い合わせ導線とサービス説明の有無を確認」）はAIの作業報告であり、読者に何も刺さらない。再定義:

> **signals = サイトから読み取った具体的事実の列挙（各≦40字・3〜4個）。「◯◯を確認」という行為の記述は禁止。読み取った中身だけを書く。**

Before: `問い合わせ導線とサービス説明の有無を確認`
After: `施工事例80件超を掲載` ／ `窓口は電話とフォームの2つ` ／ `採用ページで施工管理を募集中`

これがUI上部の「読み取り: …」帯（ChatStage L650）に出る。**3案を読む前に「うちのサイトを本当に読んだ」が成立する**——診断全体のBarnum効果はこの帯が最初に作る。

### 2.3 3軸の業界別再解釈 — 「効く一点」を当てにいく

top_line/bottom_line/fdeの3軸は維持するが、**軸の意味をその事業モデルで再解釈してから書く**よう指示する。プロンプトに次の圧縮版プレイブックを埋め込む:

| 事業モデル | top_line の効く一点 | bottom_line の効く一点 | fde の効く一点 |
|---|---|---|---|
| 施工・現場系（建設/設備） | 見積もり前の概算対応・事例資産の営業活用 | 日報/写真/書類の転記削減 | 現場↔事務所の情報の一元化 |
| 医療・介護 | 予約導線・初診前の不安解消 | レセプト・書類・日程調整 | 記録データの基盤化と共有 |
| EC・小売・多店舗 | レビュー/接客データの販促転用 | 受注・在庫・問い合わせの定型対応 | チャネル横断のデータ統合 |
| 士業・専門サービス | 相談前の「聞いていいか分からない」の解消 | 書類ドラフト・定型調査 | ナレッジの属人解消 |
| 製造・BtoB | 技術問い合わせの一次対応・提案書 | 見積もり・図面・帳票の処理 | 基幹データとAIの接続 |
| IT・SaaS | リードの一次選別・デモ前対応 | サポート・社内ナレッジ | 顧客データ分析の内製化 |
| 宿泊・観光 | 予約前質問・多言語対応 | 予約変更・定型連絡 | 口コミ/料金/稼働データの統合 |

さらに**3案の差別化規則**を明記: 「3案は同じ打ち手の言い換え（3種類のチャットボット）になってはならない。**それぞれ異なる業務面**（売る場面/繰り返しの場面/データと現場の場面）を撃つこと」。

### 2.4 few-shotの全面差し替え（P1の根治）

現行の汎用例文を削除し、**架空と明示した具体例1つ**に差し替える。例はモデルが真似る見本なので、三段構造・固有語込み・実寸の文章量で書く:

```json
{
  "companyName": "サンプル工務店（架空の例）",
  "analyzedSummary": "注文住宅とリフォームの2本柱で、施工事例を80件以上公開されているとお見受けしました。一方で窓口は電話とフォームのみ——事例で温まった見込み客を営業時間の外で取りこぼしていないでしょうか。",
  "signals": ["施工事例80件超を掲載", "窓口は電話とフォームの2つ", "採用ページで施工管理を募集中"],
  "proposals": [
    { "axis": "top_line",
      "title": "施工事例80件を、見積もり前の相談窓口に変える",
      "body": "事例を読み込んだ訪問者ほど「概算だけ知りたい」段階にいます。事例の文脈を引き継いで条件を聞き取るAI窓口を置けば、電話をためらう層や夜間の見込み客を逃しにくくなります。",
      "rationale": "施工事例の厚さと、窓口が電話・フォームのみである点から。" },
    { "axis": "bottom_line",
      "title": "現場写真と日報の整理を、AIに引き取らせる",
      "body": "施工管理を募集されている今は、現場と事務の往復が最も重い時期ではないでしょうか。写真の仕分け・日報の転記・発注書類の下書きをAIに寄せ、人の時間を現場判断に返します。",
      "rationale": "採用ページの募集職種から、現場側の人手逼迫を仮説。" },
    { "axis": "fde",
      "title": "問い合わせと施工記録を、ひとつのデータ基盤にまとめる",
      "body": "電話・フォーム・現場記録が別々の場所に残ると、次の一手が経験頼みになります。散らばった記録をAIが読める一つの基盤に集め、誰でも過去案件を引ける状態を作ります。",
      "rationale": "窓口と記録が複数に分かれている構造から。" }
  ],
  "reportTeaser": "課題を1つ選んでいただければ、御社の体制に合わせた進め方まで具体化します。"

```

※ `rationale`は現在UI非表示（§4.4で扱う）。「なぜ御社でこれか」の観測根拠1文に用途を固定する。

### 2.5 文章量・言い切り規律

- title ≦30字・**体言止めか動詞終止で言い切る**。「〜の可能性があります」をタイトルに入れない
- body 2文・計80〜130字。**1文目に観測（言い切り）、2文目に打ち手と仮説**
- hedge（〜ではないでしょうか/〜とお見受けします）は**各proposalに最大1回**。全部hedgeすると自信がなく読め、全部言い切ると捏造リスク——観測=言い切り/内情=仮説形/提案=言い切り、の層別を守る
- temperature 0.35・max_tokens 1800 は維持（質はプロンプトで作る。温度を上げて具体性を買うと捏造リスクを買う）

## 3. FocusPlan（deepen）の質 ＋ honkoma柱の織り込み

### 3.1 P2の根治 — 汎用フォールバックを例として渡すのをやめる

`example: fallbackFocusPlan(...)` を削除し、**架空明示の良例few-shot 1つ**（§3.6のafter例をそのまま使う）に差し替える。これが単独で最大の改善になる。

### 3.2 stepsの新規律 — phaseは期間、actionは固有名詞

v2正典 §2.3は本来 steps を「`1〜2週目`/`3〜6週目`/`その後`」と定めていた。現行実装は「整理/試作/運用化」という**内容ラベルに退化**し、これが汎用感の温床になっている。確定:

- **phase = 期間ラベル固定**: `1〜2週目` / `3〜6週目` / `その後`。**validatorで決定論的に上書きする**（モデルに書かせない）。期間は目安であることをriskNoteの定型で担保（「期間は目安です。初回は範囲を絞って始めます。」）
- **action = その会社の業務・資産の固有名を含む具体アクション**（≦80字）。禁止汎用語リストをプロンプトに明記: `現状把握・現状分析・最適化設計・運用定着・整理・試作・運用化・棚卸し(単独使用)・小さく始める・ヒアリング(単独使用)`。※このリストは**プロンプト指示＋再試行トリガ（soft）**とし、lint(hard)には入れない——hard化するとフォールバック（それ自体を書き直すまで汎用語を含む）が無限に落ちる

### 3.3 deepenへの燃料補給（P4の根治）

user contentに追加で渡す:
- `signals`（§2.2の新definition済みhit群。**DeepenBody型には既にある——組み立てに含めるだけ**。フロントがsignalsを送っているかChatProvider側の確認も受入基準に含む）
- 指示: 「stepsのうち最低1つは、signalsまたはanalyzedSummaryに含まれる固有の観測を引用すること」
- `painPointRaw`活用の明文化: 「painPointRawがある場合、restatementはその**語彙をそのまま再利用**して言い換えること」（自分の言葉が返ってくる＝最強の「聞いてもらえた」感。現在は渡すだけで指示がない）

### 3.4 honkoma柱の織り込みマップ（対応表・正典）

deepen system promptに4本柱を圧縮ブロックで常駐させる:

> honkomaの提供価値（提案の背骨）: ①散らばったデータをAIが読める一つの基盤に整える ②全員がAIでその基盤に触れられるようにする ③バックオフィスと繰り返し業務はAIに任せる ④人は判断・交渉・創造など人にしかできない仕事に集中する。進め方プランは必ずこのいずれかに接続する。

**pain × 柱の対応表**（プロンプトに埋め込む・fallbackの分岐にも同じ表を使う）:

| pain | 主柱 | 副柱 | プランへの織り込み方 |
|---|---|---|---|
| data（データが活かせていない） | **①データ基盤** | ② | 「どこに何が散らばっているかの特定→AIが読める基盤への集約→全員がAIで引ける状態」を steps の背骨にする |
| sales（売上・集客） | ②→① | — | 接客/営業の一次対応AIを入口に。**複数媒体・複数店舗が観測されたら①データ基盤（媒体横断の数値統合）を必ずstepsか prerequisitesに入れる** |
| support（問い合わせ・顧客対応） | **③BOAI** | ① | 一次対応のAI化。前提として「過去の問い合わせログ・FAQをAIが読める形に整える」＝①を prerequisites に明示 |
| backoffice（バックオフィス業務） | **③BOAI** | ④ | 経理・請求・日程調整・書類の具体名で。「空いた時間を何に使うか」＝④で締める |
| staffing（人手不足・属人化） | **④人は人** | ① | 属人ノウハウのデータ化（①）→繰り返し部分のAI移行→**人を判断と現場に返す（④）**を物語にする |
| other | ②共通基盤 | — | painPointRawの語彙で軸を立て、4柱のうち最も近いものに接続 |

**データ基盤提案の発火条件（社長要望の中核・明文ルール）**: painがdataのとき**および**、signals/analyzedSummaryに「複数媒体・複数店舗・複数窓口・紙やExcelでの管理・記録の分散」のいずれかが観測されているとき、**painが何であってもデータ基盤整備をsteps または prerequisites に1回、明示的な提案として入れる**。これはhonkomaが実際にやる骨太の仕事であり、「ツール1個入れて終わり」との差別化点そのもの。

### 3.5 agendaとriskNoteとcaseConnection

- **agenda = 当てにいく質問形に統一**（§1.3の型）。「最初にAIへ任せる業務範囲」のような名詞句ではなく、「見積もり前の概算質問は、今どなたが答えていますか」——バトン（壁打ち）で何が起きるかの予告編として機能し、質問自体がhitになる
- **riskNote定型**: 「期間は目安です。初回は◯◯（そのプランの第1步の対象）に範囲を絞って始めます。」——言い切りプランの誠実性担保
- **caseConnection**: 「近いです」禁止。**何が同じか**（課題の入口＋規模）を1つ言う: 「複数媒体の問い合わせが属人化していた点が、御社の状況と重なります。」（事例本文はDB原文のまま・現行規則不変）

### 3.6 before → after（FocusPlan・pain=support、工務店の例）

**Before（現行実出力レベル）**
> restatement: 問い合わせ・顧客対応を起点に、一次対応と定型確認をAIに寄せる
> steps: 整理／問い合わせ内容を分類し、AIが回答できる範囲と人が見る範囲を分けます。→ 試作／Slack、Notion、フォームなど既存導線に合わせて小さな試作を作ります。→ 運用化／担当者の確認点と改善サイクルを決め、使われる状態まで調整します。

**After（新規律準拠）**
> restatement: 「電話が現場作業を中断させる」問題を、見積もり前の概算対応から解きほぐします
> steps:
> - **1〜2週目**／電話とフォームに来た直近の問い合わせを種類分けし、「概算・仕様・日程」のうちAIが先に受けられる質問を特定します
> - **3〜6週目**／施工事例80件を回答の裏付けに使うAI一次窓口をサイトに実装し、電話の記録もまとめて一つのデータ基盤に集めます
> - **その後**／蓄積した問い合わせデータを営業と現場で共有し、次にAIへ移す業務（書類・日報）を一緒に決めます
>
> agenda: 「概算だけ知りたい」電話は週にどれくらいありますか／過去の問い合わせ記録は今どこに残っていますか
> riskNote: 期間は目安です。初回は見積もり前の概算対応に範囲を絞って始めます。

**スキーマ改訂は不要**（1.0のまま全て実現可能）。唯一の挙動変更はphaseのvalidator固定＝§3.5コントラクトの範囲内。

## 4. 「読ませる」構成

### 4.1 読む順序の設計 — 先頭3秒で「うちだ」を作る

読者が読むか捨てるかを決めるのは analyzedSummary の1文目。優先順位:

1. **1文目 = 言い当て**（社名・事業構成を具体で言い切る。「〜とお見受けしました」）
2. **読み取り帯（signals）= hitの証拠**（§2.2。要点の羅列であって作業報告ではない）
3. **2文目 = いちばん刺さりそうな一点の指名**（3案のうちどれが本命かをsummary末尾で示唆する——「特に◯◯の取りこぼしが気になります」。3案並列の「どれでもどうぞ」感を消し、言い切りの主体性を出す）
4. 3案カード → Q1チップ（現行動線のまま）

### 4.2 長さと言い切り/hedgeの配分（全出力共通の文章規律）

- **1文1アイデア**。読点3つ以上の文は分割
- summary 2文・proposal body 2文・step action 1文。**「読む気にならない」の主因は長さではなく、読んでも自社の情報が増えないこと**——具体が入っていれば現行の文量で足りる。増やさない
- hedge配分: **観測=言い切り／内情の推測=仮説形（各ブロック最大1回）／honkomaの行為=言い切り**。「〜できる可能性があると考えられます」型の二重hedgeは禁止

### 4.3 3案カードの中身の見せ方（器=ChatStageのまま）

- titleに固有語（相手の商材・業務名）を入れる——カード斜め読みでも「うちの話」と分かる
- 3案が異なる業務面を撃つ（§2.3差別化規則）——並べて読んだとき「多面的に見られた」感になる
- FocusPlanViewのstepsは期間ラベル＋固有アクション（§3.2）——「プランをもらった」感は期間の言い切りが作る

### 4.4 rationaleの死蔵問題（要1行実装・推奨）

`ProposalCard`（ChatStage L397-409）は`rationale`を**表示していない**。rationaleを「サイトのどこからそう読んだか」の観測根拠1文に用途固定（§2.4）した上で、カード下部に小さく表示することを推奨する（例: 「— 採用ページの募集職種から」）。**観測根拠の開示はBarnum効果の増幅器**（占い師が「手相のここに出てる」と言うのと同じ）で、実装は1行。器の変更ではなく中身の見せ方の範囲と判断するが、最終判断は要判断#5。

### 4.5 before → after（読み始めの体験）

**Before**: 「example.co.jp の公開情報をもとに、問い合わせ・集客・現場業務の改善余地を仮説として整理しました。」＋読み取り「問い合わせ導線とサービス説明の有無を確認」→ **自社の情報がゼロ行。ここで離脱**

**After**: 「◯◯工務店様は、注文住宅とリフォームの2本柱で…（言い当て）…特に、事例で温まった見込み客の取りこぼしが気になります（本命の指名）」＋読み取り「施工事例80件超を掲載／窓口は電話とフォームの2つ／採用ページで施工管理を募集中」→ **3行すべてが「うちのこと」**

## 付記: 自己批判と割り切り

- **固有語必須ルールは、情報の薄いサイト（1枚ペラのコーポレート）で成立しない。** その場合の観測は「サイトに◯◯の記載がないこと」自体（料金表がない・窓口が1つ等）を使う——「ない」も観測である。それでも薄ければanalysisFailed系の誠実注記に落ちる。既存動線どおり
- **業界プレイブックは7分類の粗さで外しうる。** 外した推定は3案の脇の甘さになるが、Q1チップの自己申告（pain）とpainPointRawが軌道修正装置として既に動線にある。プレイブック拡充は事例DB（RAG）が15件を超えたタイミングで再投資する方が費用対効果が高い
- **specificityはトークン消費とレイテンシを微増させる**（プロンプト増・出力同等）。max_tokensは据え置きでよく、体験上の差は誤差と判断
