# 導入事例インテークフォーム＋掲載パイプライン 設計正典

> 設計: Fable 5。実装: Opus。スコープ: **設計のみ**（Google Form接続・コード実装は後段）。
> 前提整合: `src/data/caseStudies.ts`（既存スキーマ）/ [site-ia-design.md](./site-ia-design.md) §3.5（3層開示）/ [case-study-conversion-design.md](./case-study-conversion-design.md)（`serviceType`・`industry` の frontmatter 追加、読了CTA）/ [lp-redesign-brief.md](./lp-redesign-brief.md) §8（監査#7/#8/#14 — 裏取り無し定量・煽りの禁止、声は本人許諾＋実発言のみ・捏造しない）。

## 0. 背骨（3行）

1. **フォームは「取材」であって「入力作業」ではない。** 顧客が15分で書けて、honkomaの編集30分で記事になる粒度に設問を刻む。記述必須は4問だけ、残りは選択と任意。
2. **事実と許諾を同じ1本のフォームで完結させる。** 内容を書き終えた直後が許諾判断のベストタイミング — 許諾セクションは必ず最後に置く。
3. **顧客の回答は素材、公開文は honkoma の編集著作。** ただし quote（声）だけは例外で原文主義 — 編集は表記整形まで、公開前の最終稿承認を必須ゲートにする。この1点がコンプラ設計全体の錨。

## 1. フォーム設問セット（Google Form想定・全22問）

構成: **6セクション、条件分岐1箇所**（実名選択時のみ表記確認セクションへ）。所要10〜15分と冒頭に明記。文体は「〜を教えてください」の取材体で統一し、記述問には必ず例文を添える（空欄恐怖の除去＝回答率対策）。

### フォーム冒頭文（そのまま使う文言案）

> いつも honkoma をご利用いただきありがとうございます。
> 御社の取り組みを、弊社サイトの「導入事例」としてご紹介させていただきたく、10〜15分ほどのアンケートをお願いしています。
> **いただいた内容はそのまま公開されません。** 弊社で記事の形に整えたうえで、公開前に必ず最終稿をメールでご確認いただき、ご承認をいただいてから掲載します。社名を出す・出さない（匿名掲載）もこのフォーム内でお選びいただけます。

### 2.2 スキーマ差分提案（最小2フィールド追加・すべて optional）

```ts
export type CaseStudy = {
  // ...既存9フィールドは変更なし...
  /** 支援の主分類。記事末尾CTAの文脈差し込み(conversion-design §3 L1)と将来の絞り込みに使用 */
  serviceType?: "ai" | "fde" | "product";
  /** 公開日(YYYY-MM)。IA §3.5の行リストUI移行時の日付・並び順に使用 */
  publishedAt?: string;
};
```

- `employeeRange` は**公開スキーマに足さない**判断もあり得るが、匿名事例の本文「従業員50名規模の〜」表現を型で担保したいので `employeeRange?: string`（表示用文字列・例:「従業員11〜50名規模」）として足すことを推奨。ただし優先度は低く、Opus判断で本文埋め込みでも可。
- optional なので**既存7件は無修正でビルドが通る**。既存件への `serviceType` 付与は conversion-design 実装と同時でよい。

### 2.3 匿名時の表示ルール（生成規則）

```
company = {業種の一般名詞}{法人格または「企業」}（{文脈語}）
  文脈語 = 事業形態 or 支援を受けた部門（例: クリニック運営／経営企画部／営業組織）
例: 医療法人（クリニック運営）／ITサービス企業（経営企画部）／広告代理店（デジタルマーケティング）
```

- 規模は company に入れず本文で「従業員◯〜◯名規模」の**帯表現**のみ。
- **特定可能性チェック**（公開前レビュー項目）: 地名・固有サービス名・「業界唯一」等の絞り込み表現を本文から除去。業種×規模×施策の組合せで1社に絞れる場合は業種を1段抽象化（「◯◯クリニック」→「医療」）。
- `role` は役職のみ・氏名なし。quote は匿名でも**本人の実発言のみ**（匿名は身元の加工であって、発言の加工ではない）。

## 3. パイプライン設計（記入→公開）

### 3.1 全体フロー（Phase 0 = 今日から回る形）

```
① 送付      honkomaがフォームURLを顧客担当者へメール（送付テンプレは§6）
② 記入      顧客がフォーム回答（10〜15分）
③ 受領      Google Formの回答通知メール → 管理スプレッドシートに1行転記(Phase0は手動)
④ 編集      honkomaが本設計§2のマッピング表どおりに CaseStudy ドラフト＋記事文を作成
             - oneLiner/tagsのコピーライティング、challenge/actions/outcomeのリライト
             - 実施記録との突合（書かれた actions を本当にやったか）
             - コンプラチェックリスト(§5.3)を通す
⑤ 最終稿確認 記事プレビュー(文面全文)をQ4のメールへ送付 →「この内容で公開して問題ありません」の
             返信を受領。修正希望があれば反映して再送（quoteの変更は必ずこのループで）
⑥ 記録      承認メールの日付・文言をスプレッドシートに記録（許諾エビデンス）
⑦ 反映      caseStudies.ts に1オブジェクト追記 → ビルド → デプロイ
⑧ 事後      公開URLを顧客へ共有(お礼)。取り下げ・修正依頼の窓口を明記
```

**自動/人手の線引き（Phase横断の原則）**: ③⑦は自動化してよい。**④のコピー編集と⑤の承認確認は永久に人手**。理由 — ④は編集著作（oneLinerの質がカードのCTRを決める）、⑤は法的ゲート（自動化した瞬間「許諾済み」の主張が崩れる）。

### 3.2 Phase 1（スプレッドシート連携・半自動取り込み）

- Google Form → 回答スプレッドシート自動連携（Formの標準機能・開発ゼロ）。
- スプレッドシートに管理列を追加: `status`（新規／編集中／顧客確認中／承認済／公開済／保留／取り下げ）、`承認日`、`承認メール控えリンク`、`公開URL`、`slug`。
- Claude Code 側に取り込みスクリプト（またはスキル）: シート1行を読み、§2のマッピング表どおりに `CaseStudy` オブジェクトの**ドラフト**と記事文候補を生成 → 人間が④⑤を実施 → 承認後にオブジェクトを caseStudies.ts へ追記する diff を出す。
- ここでも生成物は常に「ドラフト」。status が `承認済` でない行から公開 diff を作らないことをスクリプトのガードにする。
- 将来（Phase 2・参考まで）: Q20で「そのまま紹介してよい」の事例は note 記事・YouTube 台本の素材プールへ。ここは本設計のスコープ外だが、フォームのQ20が既にその入口になっている。

### 3.3 管理スプレッドシートが正・caseStudies.ts は公開ビュー

許諾状態・連絡先・承認記録の SSOT はスプレッドシート。caseStudies.ts には**公開してよい情報しか存在しない**状態を保つ（リポジトリは公開物と同じ扱い）。取り下げ依頼が来たら: シートを `取り下げ` に更新 → ts からオブジェクト削除 → デプロイ、の3手で完結。

## 4. /case-studies への組み込み

- **箱は既存のまま使う。** 新規事例は `caseStudies` 配列への1オブジェクト追加だけでカード（一覧）と記事（詳細）が同時に生える — 現行の `CaseStudiesPage` / `CaseStudyDetailPage` は変更不要。quote が無ければ声ブロックが出ない・logo が無ければ社名テキスト表示、という**既存のフォールバックがフォームの任意設問とそのまま対応している**（これがこのフォーム設計の根拠でもある）。
- **並び順**: 実名（logo付き→logoなし）→ 匿名、の現行秩序を維持。件数が8件を超えたら `publishedAt` 降順＋実名優先へ移行（IA §3.5 の行リストUI刷新と同時でよい）。
- **"声"プレースホルダの差し替え運用**: 貞栄会・慶洋・広告代理店（AViC）の「quote: 取引先Q&A回収後に追記」は、**このフォームの最初の送付先**にする（＝パイロット3社）。既掲載先には全設問でなく、**セクション5〜7だけの短縮版リンク**（Google Formのセクション構成なら同一フォームで冒頭に「すでに事例掲載中の方はこちら」分岐を作れる。運用が煩雑なら別フォーム1本でも可 — 推奨は分岐1本化）。回収後は `quote`/`role` の2フィールド追記のみのライトパス（⑤最終稿確認は声部分のみで実施）。
- **AViC の実名化**: 現在 `adagency-anon`。フォームQ18で「実名可」が返れば `named: true`・社名・slug変更（旧slugからの `<Navigate replace>` を1本置く）。この昇格パスも⑤の承認を経る。
- **読了→問い合わせ動線との接続**: 新規事例が `industry`（必須）と `serviceType` を持つことで、conversion-design §3 の L1 コピー差し込み（「{industry}のこの事例と同じ観点で〜」）が全件で機能する。**記事本文にCTAを混ぜない原則（同 §1-2）はフォーム由来の事例でも同一** — 顧客の声の直後に売り込みを置かない。

## 5. コンプラ/許諾の設計

### 5.1 三重ゲート

| ゲート | 場所 | 担保するもの |
|---|---|---|
| ① フォーム同意（Q19の4チェック） | 記入時 | 事実性・定量の自己裏取り・最終稿確認への同意・撤回権の認知 |
| ② 最終稿承認（メール返信） | 公開前 | **具体的な公開文面への許諾**（フォーム同意は「掲載一般」への同意にすぎない。文面承認が本丸） |
| ③ 承認記録（スプレッドシート） | 公開後 | 「いつ・誰が・どの文面に」承認したかのエビデンス。問い合わせ・監査に即答できる状態 |

### 5.2 quote（声）の編集規律

- **可**: 誤字修正・敬体の統一・明らかな脱字補完（＝表記整形）。
- **本人再確認が必要**: 語順変更・短縮・複数回答の結合（⑤のループで「この形に整えました。趣旨が変わっていないかご確認ください」と明示して承認を取る）。
- **禁止**: 言っていない語の追加・成果の増幅（「減った」→「激減した」等）・別人の発言との合成。
- 空欄なら quote なしで公開（既存運用と同じ）。**「声が無いと事例にならない」という圧をかけない**こと自体が捏造防止の設計。

### 5.3 定量成果の掲載基準（監査#7の運用化）

```
掲載可 = Q14に記載あり ∧ Q15で確認手段の申告あり(「その他」の場合は内容が具体的) ∧ 最終稿で本人承認
それ以外 → outcomeは定性表現のみ（「大幅に」「目に見えて」等、既存事例のトーン）
```

- 数値を載せる場合も**過去の事実として**書く（「◯◯時間削減した」）。「御社も◯％削減」型の約束転用は禁止（conversion-design §4 の禁止リストと同一線）。
- 「最短◯週間」「◯社中◯位」等の優良誤認になり得る比較・速度表現は、フォーム回答にあっても掲載しない（監査#11/#14）。

### 5.4 匿名化の担保

- Q18で匿名選択時は §2.3 の生成規則を機械的に適用し、⑤の最終稿確認で**「この表記で御社と特定されないか」を顧客自身にも確認してもらう**（特定可能性の最終判定者は本人）。

## 社長への要判断リスト

1. **匿名の既定**: フォームで「相談したい」やQ18無回答の場合、匿名に倒して先に進めるか、掲載保留にするか。**推奨: 掲載保留**（許諾のグレーを作らない）。
2. **最終稿承認の形式**: メール返信で足りるか、freeeサイン等の記名承諾まで求めるか。**推奨: メール返信＋スプレッドシート記録**（上場企業案件のみ先方の広報承認プロセスに従う）。
3. **二次利用（note/YouTube）の設問（Q20）を初版から入れるか**。**推奨: 入れる**（既定「媒体ごとに確認」なら顧客負担ゼロで将来の資産化入口が確保できる）。
4. **自動化の上限**: 「最終稿承認は必ず人間・公開の完全自動化はしない」を恒久ルールとして確定するか。**推奨: 確定する**（本設計はこれを前提に組んである）。
5. **パイロット送付先**: 貞栄会・慶洋・AViC の3社から始めることの承認（既掲載で関係が温かく、quote回収・実名昇格の実益が即出る）。
6. **謝礼の有無**: 回答インセンティブ（公開記事の営業利用許可・Amazonギフト等）を付けるか。**推奨: 金銭謝礼なし**（謝礼付き推薦は声の中立性を毀損し、ステマ規制の照会リスクを増やす。お礼は公開URL共有と記事の先方利用許可で代替）。
